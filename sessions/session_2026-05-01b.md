### 1. Rate uncertainty and dual-global blending

`compute_sales_rates` now returns `(rate_lookup, global_rates_short, global_sigma_short)`.

Two global windows: 30d and 90d. Blending weights:
- high (N_s ≥ 10): 0.80·short + 0.20·long
- medium (5 ≤ N_s < 10): 0.50·short + 0.20·long + 0.20·global_30d + 0.10·global_90d
- low (0 < N_s < 5): 0.25·short + 0.20·long + 0.35·global_30d + 0.20·global_90d
- no_recent (has 90d): 0.25·long + 0.45·global_30d + 0.30·global_90d
- no_recent (no data): 0.60·global_30d + 0.40·global_90d

Poisson uncertainty: `σ_rate = √N / T`. Propagated through blend: `σ_blend = √(Σ wi²σi²)`. Each rate_lookup entry now includes `rate_sigma`.

### 2. Sellout date range

For each product, `build_report_data` computes:
- `rate_high = max(blended, global_30d)`, `rate_low = min(blended, global_30d)` with respective sigmas
- `sellout_early`: stock / (rate_high + σ_high) — worst case
- `sellout_late`: stock / max(rate_low − σ_low, ε) — best case, capped at 365 days
- `est_days_early` / `est_days_late`: the corresponding day counts

The spread captures both statistical noise and systematic bias (different customer bases pooled in global rate).

### 3. Restock event auto-detection (`load_restock_events`)

Parses `docs/inventory_history.csv` (written every 30min by scrape-prices.yml) to detect restock events. Threshold: ≥3 products with positive deltas, total delta ≥5 units at same (timestamp, machine).

Returns:
- `last_restock_dates`: {machine → date} — replaces hardcoded `LAST_RESTOCK_DATES` (still used as fallback for Chicago machines not yet in history)
- `restock_stock_targets`: {(machine, product) → new_stock at last restock} — used as fill target instead of API capacity field

Detection constants: `MIN_RESTOCK_PRODUCTS = 3`, `MIN_RESTOCK_UNITS = 5` in config section.

### 4. Visit-projected fill quantities

`build_report_data` computes `days_until_visit = max(0, (next_visit − today).days)` per machine, where `next_visit = last_restock + freq`. Fill = `max(0, fill_target − max(0, stock − daily_rate × days_until_visit))`. Accounts for sales between today and the next visit.

`physical_capacity = max(restock_stock_targets.get(..., 0), MIN_SLOT_CAPACITY=16)`, further raised by API capacity if it's larger. The minimum of 16 reflects that slots can always hold at least that many, and popular products can double up.

## How the script works

Runs in three modes, controlled by `--mode`:

- **`auto`** (default in CI): Monday ŌåÆ sends weekly summary (7 days ending yesterday); other days ŌåÆ checks yesterday only and sends a daily alert if >10 items sold; otherwise silent.
- **`weekly`**: always fetches MonŌĆōSun (7 days ending yesterday) and sends.
- **`daily`**: always fetches yesterday and sends (ignores threshold).
- **`--days N`**: manual override, looks back N days from today, always sends. Useful for testing.
- **`--mock`**: skips all API calls, uses hardcoded fake data (see `_MOCK_ORDERS_TEMPLATE` and `_MOCK_INVENTORY`). No credentials needed. The mock data clamps all day offsets to the date range, so a 1-day range puts 20 items on one day (triggers the alert) and a 7-day range spreads them across the week.

```bash
# preview email layouts locally ŌĆö no credentials needed:
python mhub_sales_report.py --mock --mode weekly
python mhub_sales_report.py --mock --mode daily
```

To also send: add `--email` and set `GMAIL_ADDRESS` / `GMAIL_APP_PASSWORD`.

In GitHub Actions: workflow_dispatch inputs include `mode`, `days`, and `mock` (set `mock: true` to preview without hitting the API).

## Email structure

Both email types (weekly and daily alert) have the same layout:
1. Branded navy banner (BLANKET BOX / mHUB Prototyping Shop ┬Ę Weekly Sales Report or ┬Ę Activity Alert)
2. Intro text with date range
3. **Sales Summary table**: Product | Units Sold | Pre-tax ŌĆö with footer rows: Total pre-tax, Sales tax (+ $X), Total collected ($Y)
4. **Transactions table**: Date/Time | Product | Qty | Pre-tax ŌĆö with a per-order subtotal row after each order group showing pre-tax + tax = total paid
5. **Inventory table** (if stock values are meaningful ŌĆö suppressed when all > 100, which is the mHUB "9999-mode" where the machine starts at 9999 and just decrements)

Daily alert subject: `"High activity day at Blanket Box ŌĆö June 30, 2026"`  
Weekly subject: `"mHUB Sales Report ŌĆö June 24ŌĆō30, 2026"`

## Key data facts

- **mHUB freezer ID**: `15520` (in `ops_scraper.LOCATION_GROUPS["mckinley"]`)
- **mHUB machine name in order exports**: `"mHUBPrototypingShop"` ŌĆö this is NOT in `restock_report.py`'s `MACHINE_NAME_MAP`, which is why mHUB orders were always excluded from the restock report's rate calculations. The new script handles it directly.
- **Revenue column**: `"Sales volume"` in the Sandstar XLSX (= Quantity ├Ś Unit price, pre-tax per product line). Not `"Total amount"` ŌĆö that column doesn't exist; `"Order amount"` is the per-order pre-tax total (used for tax computation).
- **Tax split**: `Order amount` = pre-tax (per order); `Payment Amount` = total paid including tax. Tax = `Payment Amount - Order amount`, computed after deduplicating on `Order number`. For mHUB (Chicago), effective rate Ōēł 9.75% (IL 7.25% + Chicago 1.25% + Cook 1.25%).
- **Row filtering**: mirrors `FullMonthlyReport.py`'s `clean_sales_df` ŌĆö drops `Payment channel == "offline"` and `Order amount == 0`. Does NOT use `Payment status == "Success"` (that column exists but isn't used in the monthly report).
- **mHUB inventory**: machine runs in "9999-mode" (stock initialized to 9999, decrements with sales). Values like 9927 still mean "not meaningfully tracked." Inventory table is suppressed when all stocks > 100. If the stockers ever start entering real stock counts, it'll appear automatically.
